7.线程的状态

7.1线程的五大状态

  1. new Thread();—>>线程对象一旦创建就进入到新生状态
  2. 调用 start()方法，线程进入就绪状态，但不意味着会立即被调度要等待cpu的调度
  3. 运行状态:进入运行状态，线程才真正执行线程体的代码块
  4. 当调用sleep，wait或同步锁定时，线程进入阻塞状态，就是代码不往下执行，阻塞事件解除后，重新进入就绪状态，等待cpu调度
  5. 死亡:–>>线程中断或者结束就进入死亡状态，一旦进入死亡状态，就不能在启动

7.2线程停止:

  • 不推荐使用jdk自带的 stop() 和 destroy() （已废弃）
  • 推荐线程自己停止
  • 建议使用一个标志位进行终止变量，当flag=false 时，则线程终止运行

7.3线程休眠

  • sleep指定当前线程阻塞的毫秒数
  • sleep存在异常InterruptedException
  • sleep时间达到后，线程进入就绪状态
  • sleep可以模拟网络延时，倒计时等
  • 每一个对象都有一个锁，sleeo不会释放锁

9 锁

死锁问题

产生死锁的原因:

  1. 系统资源的竞争；进程推进顺序非法；死锁产生的必要条件。
  2. 一个资源每次都只能被一个进程使用（互斥）
  3. 一个进程因请求资源阻塞时，对已获得的资源保持不放（请求与保持条件）
  4. 进程已获得的资源在未使用完之前，不能强行剥夺
  5. 若干进程之间形成一种头尾相接的循环等待资源关系

如何避免死锁问题

只需要想办法破除任意一个或多个产生死锁的条件，就可以避免死锁的发生

Lock(锁)

  1. JDK5.0开始，java提供了线程同步机制，通过显示的定义同步锁对象来实现同步
  2. 每次只能有一个线程对Lock对象加锁，线程开始访问共享资源之前应先获得Lock对象
  3. ReentrantLock类实现了Lock，它拥有synchronized相同的并发现和内存语义，在实现线程安全的控制中，比较常用的是ReentrantLock，可以显示的加锁，释放锁

Lock 与 synchronized 的区别

  1. Lock是显示锁(手动开关) synchronized是隐式锁,处了作用域自动释放
  2. Lock只有代码块锁,synchronized有代码块锁和方法锁
  3. 使用Lock锁，JVM将花费较少的时间来调度线程，性能更好，并且具有更好的扩展性(提供更多的子类)
  4. 优先选择顺序:Lock > 同步代码块(已经进入方法同体，分配了相应资源) >同步方法(在方法体之外)

11.线程池

使用线程池

  1. JDK5 开始 提供了 线程池相关API : ExecutorService 和 Exceutors
  2. ExecutorService : 真正的线程池接口 常用子类ThreadPoolExecutor
    1. void execute(Runnable command):执行任务/命令,没有返回值，一般用来执行Runnable
    2. Futuresubmit(Callable task) :执行任务,有返回值，一般用来执行Callable
    3. 使用void shutdown()关闭线程池
  3. Executors: 工具类，线程池的工具类 ，用于创建并返回不同类型的线程池
